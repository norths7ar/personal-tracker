# ROADMAP

这个路线图记录 personal-tracker 后续开发方向。当前项目已经具备本地 SQLite 存储、Streamlit 页面、开销记录、饮食记录、流水编辑、导出和基础分析能力。后续重点不只是新增页面，而是降低长期记录成本、提高数据质量，并让积累的数据能产生更明确的洞察。

## 核心判断

- 这个项目的核心价值不是“能记”，而是“低负担地持续记”。
- 后续优先级应围绕三个问题排序：
  - 是否能跨设备访问，离开主 PC 后还能不能及时记录？
  - 录入是否足够快，能不能每天自然使用？
  - 数据是否足够干净，几个月后还能分析？
  - 分析是否能回答真实问题，而不只是展示汇总图？

## 近期优先级：跨设备同步可用

### 单用户云端版

将 personal-tracker 从单机本地应用升级为跨设备可访问的单用户云端工具。目标是让 PC、笔记本和手机浏览器都能访问同一份数据。

推荐路径：

- 前端继续使用 Streamlit。
- 数据库从 SQLite 迁移到托管 PostgreSQL，优先考虑 Supabase 免费层。
- 部署到 Streamlit Cloud 或同类轻量托管平台。
- 增加单用户访问保护，至少使用密码或 token，不裸露个人财务和饮食数据。

暂不推荐路径：

- 不把云盘同步 SQLite 作为主同步方案。它适合作为备份，但不适合作为跨设备并发访问的主数据库。
- 不在第一阶段引入完整账号体系，除非后续明确需要多用户。

价值：

- 离开主 PC 后仍然可以记录。
- 手机浏览器可以直接使用，不需要先开发 app。
- 为后续移动端优化、批量录入和自动化入口打基础。

### 数据访问层抽象

在迁移 PostgreSQL 前，先把页面和数据库实现之间的边界整理清楚。

目标能力：

- 页面继续调用稳定的数据函数。
- 底层支持 SQLite adapter 和 PostgreSQL adapter。
- 本地开发可以继续使用 SQLite。
- 云端部署使用 PostgreSQL。
- 数据库连接信息来自环境变量或 Streamlit secrets。

价值：

- 降低一次性迁移风险。
- 保留本地开发和离线测试能力。
- 后续如果引入 FastAPI，可以复用同一层数据访问逻辑。

### SQLite 到 PostgreSQL 迁移

提供一次性迁移脚本，把本地 SQLite 历史数据导入 PostgreSQL。

目标能力：

- 迁移 transactions、diet_meals、diet_foods。
- 保留原始 id、日期、金额分、分类、备注、置信度和创建时间。
- 迁移前检查目标库是否为空或要求明确确认。
- 迁移后输出行数对账。

价值：

- 避免云端版从空数据开始。
- 让切换主数据库可验证、可回滚。

### FastAPI 第二阶段

FastAPI 是有价值的后续方向，但不作为第一阶段跨设备访问的必要条件。

适合引入 FastAPI 的信号：

- 需要真正的移动端 app。
- 需要 CLI、快捷指令、机器人或其他自动化入口。
- 需要多个前端共用同一套业务逻辑。
- 需要更完整的账号体系和 API 鉴权。
- Streamlit 页面中的业务逻辑已经明显过重。

价值：

- 作为后端练手场景很自然。
- 为未来多入口、多客户端架构做准备。

## 近期优先级：降低记录成本

### 批量自然语言录入

支持一次输入多条自然语言记录，并自动拆分为多条开销或饮食记录。

示例：

```text
今天早餐豆浆包子，中午麦当劳 35，晚上超市买菜 86，打车 24
```

目标能力：

- 自动识别多条事件。
- 自动区分开销和饮食记录。
- 对金额、日期、餐顿、分类、食物清单进行结构化提取。
- 进入批量确认页，允许逐条编辑、删除、保存。

价值：

- 显著减少每天记录的操作次数。
- 比单纯新增分析图更能提升使用频率。

### 常用模板和快捷复用

为高频记录提供一键复用能力。

候选功能：

- 最近记录再次保存。
- 常用开销模板，例如“公司食堂午餐”“停车费”“停车月卡”。
- 常用饮食模板，例如“早餐豆浆包子”“公司午餐”。
- 根据历史记录自动推荐常用项。

价值：

- 减少重复输入。
- 为没有必要调用 LLM 的场景提供更快路径。

### 移动端录入体验

优化 Streamlit 在手机浏览器上的输入流程。

候选改进：

- 默认日期为今天。
- 录入完成后快速继续下一条。
- 最近分类优先。
- 金额输入更适合移动端。
- 减少确认页上的非必要信息。

价值：

- personal-tracker 的使用场景很可能发生在消费或吃饭后，手机端体验直接影响记录完整度。

## 中期优先级：数据质量和可维护性

### 历史分类记忆

用户手动确认或修改分类后，系统记录相似描述和分类之间的关系。之后遇到相似描述时，优先使用历史分类，再考虑调用 LLM。

候选实现：

- 维护 keyword、merchant 或 normalized_description 到分类的映射表。
- 对高频描述形成规则，例如“麦当劳 -> 餐饮 / 堂食或外卖”。
- 在低置信度时优先展示历史候选。

价值：

- 降低 LLM 调用次数。
- 提高分类一致性。
- 减少用户重复校正。

### 低置信度待处理队列

将 LLM 低置信度、未知分类、抽取失败、空字段等记录集中展示。

候选检查项：

- 开销记录：未分类、空子类、低置信度、金额异常、疑似重复。
- 饮食记录：空食物名、低置信度、餐顿类型异常、疑似重复。
- 支持批量确认、批量修改和忽略。

价值：

- 避免数据问题散落在不同页面。
- 让维护数据质量变成一个明确工作流。

### 分类管理页面

将 `config.yaml` 中的分类管理逐步迁移到 UI。

候选能力：

- 新增、重命名、删除主分类和子分类。
- 合并分类。
- 批量更新历史记录。
- 显示分类使用频率，帮助清理长期不用的分类。

价值：

- 降低维护门槛。
- 避免手动编辑配置文件带来的错误。

## 中期优先级：更有用的分析

### 预算和预警

支持按月设置总预算和分类预算。

候选能力：

- 月总预算。
- 分类预算，例如餐饮、交通、购物。
- 已用比例和预计月底花费。
- 超预算或接近预算提醒。

价值：

- 从“记录过去”扩展到“管理当下”。

### 趋势解释

在现有图表基础上生成简短文字解释。

示例：

```text
本月支出比上月增加 420 元，主要来自餐饮和交通；其中外卖增加 260 元。
```

实现思路：

- 先用规则生成可解释摘要。
- 后续可选接入 LLM，将结构化统计转成更自然的中文说明。

价值：

- 降低读图成本。
- 让分析页直接回答“为什么变多/变少”。

### 饮食维度升级

当前饮食分析偏记录覆盖率和食物频率，后续可以加入更有实际意义的标签。

候选维度：

- 外食、在家、外卖。
- 饮料、零食。
- 粗粒度营养结构：蛋白质、蔬菜、水果、主食。
- 一周饮食结构回顾。

价值：

- 从“我吃了什么”升级到“饮食结构是否健康、是否规律”。

## 后续优先级：导入、备份和迁移

### 自动备份

为本地 SQLite 数据库增加自动备份机制。

候选策略：

- 每次启动时备份。
- 每天最多备份一次。
- 保留最近 N 份备份。
- 备份文件放在 `data/backups/`，继续不进入版本控制。

价值：

- 本地工具的数据安全非常关键。
- 防止误删、迁移失败或本地文件损坏。

### CSV 导入

在已有 CSV 导出的基础上补充导入能力。

候选能力：

- 通用 CSV 字段映射。
- 预览导入结果。
- 重复记录检测。
- 后续支持支付宝、微信、银行账单等特定格式。

价值：

- 让历史数据进入系统。
- 降低从其他记账工具迁移的成本。

### 数据修复脚本

提供面向维护的脚本或页面。

候选能力：

- 重新计算金额分字段。
- 重跑分类。
- 合并重复食物名。
- 修复空字段或异常字段。
- 执行 schema 迁移检查。

价值：

- 支持长期迭代。
- 降低未来结构调整的风险。

## 可能的新领域

在开销和饮食稳定后，可以考虑扩展到更多个人追踪方向。

候选方向：

- 运动记录。
- 睡眠记录。
- 体重记录。
- 情绪记录。
- 药物或健康事件记录。

这些方向不应过早加入。更合理的前提是：当前开销和饮食两条主线已经形成低摩擦录入、可靠数据质量和有用分析闭环。

## 建议执行顺序

1. 单用户云端版：Streamlit 部署 + PostgreSQL + 简单访问保护。
2. 数据访问层抽象：保留 SQLite 本地开发，新增 PostgreSQL 云端运行。
3. SQLite 到 PostgreSQL 迁移脚本。
4. 批量自然语言录入。
5. 历史分类记忆。
6. 低置信度待处理队列。
7. 预算和异常分析。
8. 自动备份。
9. CSV 导入。
10. 分类管理页面。
11. 饮食维度升级。
12. FastAPI 后端拆分。
13. 运动、睡眠、体重、情绪等新领域。
